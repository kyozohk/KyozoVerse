# KYPRO-61

Actual result: The backend saves the raw community name verbatim as the URL slug (e.g. href="/!@#$%^&*()+=[]{}|;':",.<>?/`~"). Clicking the community card triggers a full application crash: "Application error: a client-side exception has occurred" with console error: URIError: URI malformed at decodeURIComponent
Assignee: Ashok
Bug title: Community with raw-name URL slug is inaccessible after creation — crash or "Community not found" depending on character type
Category: Functional
Environment: http://pro.kyozo.com – Chrome
Expected result: The community should either not be created with an invalid URL, or a safe fallback slug should be used. Navigating to the community should work without errors
Notes: Three distinct failure modes observed:

1. NON-LATIN UNICODE (Cyrillic, Chinese — likely all Unicode scripts):
   Examples: "клавдий тестирует" → /клавдий-тестирует, "你好" → /你好
   Browser navigates to percent-encoded URL (e.g. /%E4%BD%A0%E5%A5%BD)
   Result: no crash, sidebar shell loads with community name visible, but main content shows "Community not found"
   Community is inaccessible

2. SPECIAL CHARACTERS containing URI-breaking chars (%, #, ?, etc.):
   Example: "!@#$%^&*()+=[]{}|;':"",./<>?`~"
   Result: full app crash — "Application error: a client-side exception"
   Console: URIError: URI malformed at decodeURIComponent
   App must be manually reloaded

3. PARTIAL SPECIAL CHARACTERS (chars that don’t break URI parsing):
   Examples: "test-new!", "[test-:)]"
   These navigate without crashing but communities may still be inaccessible

Root cause: empty handle (see KYPRO-60) causes backend to store raw community name as slug. Pattern is consistent across all tested non-Latin scripts.
