# KYPRO-70

Actual result: Attendees who purchased multiple ticket types are imported as separate, duplicate audience entries — one per ticket type. For example, an attendee registered for both 'General Admission' and 'Second category' appears twice in the audience with the same name and email. In a test with 16 Eventbrite rows (13 unique attendees), all 16 rows were imported, resulting in 3 duplicate audience entries.

Further investigation confirmed the impact goes beyond the Audience list view: each duplicate row creates a genuine duplicate community membership. Affected profiles (testuser2, testuser9, testuser11) all show 'Member of 2 Communities' — both entries being the exact same community (t1 @t1). This means duplicated users will receive broadcasts twice, count twice in member stats, and appear twice in guestlists.
Assignee: Ashok
Bug title: Eventbrite import creates duplicate audience entries for attendees with multiple ticket types
Category: Functional
Environment: Production
Expected result: Each unique attendee (identified by email address) should appear as a single audience entry. If they purchased multiple ticket types, this could be represented via tags or notes, but they should not be duplicated as separate members.
Notes: Confirmed during happy path testing of Eventbrite import on community t1. Duplicates observed: mailto:testuser9@gmail.com (x2), mailto:testuser11@gmail.com (x2), mailto:testuser2@gmail.com (x2). The ticket type is stored in the Notes column in the preview — deduplication by email with merged notes would be the expected behaviour.

Follow-up check on member profiles confirmed each duplicate Audience row results in a separate community membership record. The 'Communities' stat on each affected profile shows 2, with 'Member of 2 Communities' listing t1 @t1 twice. This elevates the bug to Critical — duplicate memberships cause real data integrity issues beyond cosmetic duplication.
