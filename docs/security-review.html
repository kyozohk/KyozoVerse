Confidential Internal review

A security review,
and the fixes already shipped.

An independent audit of KyozoVerse — secrets, authentication, database rules, API surface, dependencies — followed by a focused engineering sprint. This document captures both the findings and the work that's now landed on the security/critical-fixes branch.

01 The headline

Most of what we found is now closed.

We started the audit expecting a long backlog. The codebase turned out to be in good architectural shape — the issues clustered in a handful of places: a stale super-admin page, database rules written for an early prototype, public API routes that had drifted past their guardrails, and a deep stack of out-of-date third-party packages. Almost all of it was closeable in a single sprint.

The remaining work is administrative rather than engineering — rotate the secrets that lived in the developer's local .env, run pnpm install to pull in the dependency upgrades, and deploy the new Firestore and Storage rules. After that, KyozoVerse is in a position where a follow-up audit ninety days from now would look very different.

The single most exploitable thing

A super-admin panel at /sa guarded by the password "kyozo123" — written into the source code, shipped to every visitor's browser, and giving its bearer the ability to read every user record and reassign any community to any other account.

The rest of the audit found things in the same shape: defaults left over from prototype days that never got tightened. The Firestore rules, for instance, said "any authenticated user can read every other user's profile" — which was fine when a user was a co-founder on dev, and a GDPR breach when the platform opened to invitees. None of these are bad designs. They're the rough edges every shipping codebase carries until someone sits down and goes through the surface area systematically.

03 What we shipped

Seven changes, one branch.

Each of these landed as part of a single commit on security/critical-fixes. The before-and-after framing below is meant for non-engineers — what it looked like to a hypothetical attacker on Monday, versus what it looks like on Friday.

01

The hardcoded admin password is gone Shipped

The /sa super-admin page and the /api/sa/* endpoints that powered it have been deleted entirely. The page lived as dead code anyway — Next.js was deploying the routes from the app/ folder, not from src/app/ where this page lived — but a hardcoded password sitting in a Git-tracked file was a footgun waiting to be re-enabled. Layout references and the eight remaining duplicate route files were also removed.

Before

A hidden URL with the password "kyozo123" hardcoded in the React bundle. Any visitor who guessed it could read every user, every community, and reassign communities between owners.

After

The page is deleted, the API routes are deleted, and the layout fallthrough that excluded the path from auth checks has been removed. Workspace administration is now accessed via the existing /settings/team page, which goes through Firebase Auth and a server-side role check.

02

Public API routes now require authentication Shipped

Eight routes were callable by anyone on the internet with a curl command. Each has been categorised by the level of trust it requires and gated accordingly: workspace-admin for waitlist approve/reject/approve-direct; any signed-in workspace user for AI generation; per-IP rate limiting on the public waitlist signup. Webhook endpoints — which can't use Bearer tokens because their callers are external services — now verify pre-shared secrets and HMAC signatures instead.

Before

Anyone could approve their own waitlist signup. Anyone could spam the AI generation endpoint and run up the Gemini bill. Anyone could POST a forged email into the inbox webhook and have it stored as if it came from a real user.

After

Each route declares the auth level it needs at the top of the handler. Inbox webhook payloads are now verified against Resend's Svix signature with a five-minute replay window. AI generation has a thirty-call-per-minute rate limit per IP.

03

The database rules were rewritten from scratch Shipped

The previous Firestore rules were written for a prototype with a handful of trusted users. They allowed any logged-in user to read every record in users, write to workspaceMembers (the platform-admin role table), and write to inboxMessages with no caller validation. The new rules deny by default and explicitly open up only the paths the application actually needs, with strict guards on what's being changed.

Before

One signup gave you a copy of every user's name, email, and phone number. One Firestore write made you a workspace admin. The rules read like an early prototype — because they were one.

After

Users can only read their own row. Workspace-membership writes are server-only via Admin SDK, with a single narrow exception for a new user claiming their pending email-based invite. Storage rules check community membership before allowing uploads and refuse SVGs, which can carry executable JavaScript.

04

New server endpoints replaced direct database access Shipped

To make the locked-down rules workable from the client, four new server endpoints were added: /api/me/role (your own platform role), /api/admin/workspace-members (team CRUD with a "can't remove the last admin" guardrail), and /api/admin/waitlist (replaces a direct Firestore listener). The team-management page and the platform-role hook were rewritten to talk through these endpoints instead of querying Firestore directly.

Before

Client code did updateDoc(workspaceMembers, ...) and the database trusted it. Any signed-in user could promote themselves to admin in one network request.

After

The same operations go through a server route that re-verifies the caller's role before mutating. The dev@kyozo.com hardcoded admin email check was also removed and replaced with the proper role-based check everywhere it appeared.

05

Email templates can no longer be hijacked for phishing Shipped

The waitlist approval and rejection emails used to interpolate the recipient's first name into the HTML template literal. A user signing up with the first name <script>…</script> could inject arbitrary content into transactional emails sent from your verified Kyozo domain — a phishing primitive even when the recipient's email client doesn't run scripts. A new shared escaping utility (src/lib/html-escape.ts) is now applied everywhere user input meets HTML.

Before

Any registration name, post title, or community handle could break out of its container in the email and inject content. The recipient saw an email from your verified domain, so they trusted it.

After

Every user-controlled value passes through escapeHtml() before it touches an HTML template. The same applies to URLs interpolated into href attributes.

06

Uploads are no longer permanent leakage risks Shipped

The /api/upload route had three problems stacked on top of each other: it generated signed URLs that expired in the year 2500, it took the uploader's user ID from a client-controlled HTTP header, and it allowed SVG uploads (which can contain inline JavaScript). All three are fixed: URLs are now public and revocable, the uploader is the verified Firebase Auth UID, and SVG is removed from the allow-list. Filenames are also sanitised against path traversal.

Before

A signed URL leaked into a log file, a screenshot, or a copy-paste was a permanent backdoor — by the time the leak was noticed, you couldn't take it back. The uploader header could be spoofed to attribute uploads to other users.

After

Public read URLs are revocable in seconds via the bucket dashboard. The uploader is whoever's Firebase token validated. SVG uploads are rejected.

07

The dependency stack is no longer a CVE pile-up Needs your install

The repository carried 26 critical and 33 high-severity vulnerabilities, almost all in transitive dependencies of genkit, @modelcontextprotocol/sdk, and other AI/MCP packages — Handlebars (code injection), Lodash (prototype pollution), fast-xml-parser, axios (SSRF), jws (signature bypass), protobufjs, and Next.js itself. The package.json now bumps Next.js and adds pnpm.overrides entries that pin all of these to their fixed versions across the entire dependency graph. Two unused dependencies were also dropped: mongo (a 16-year-old abandoned package) and sendgrid (the deprecated v5 — the project uses @sendgrid/mail and Resend).

Before

Every npm install brought in known-exploitable code. Several of the CVEs have public exploits that enable remote code execution or authentication bypass.

After

Once pnpm install runs against the new package.json, the override entries force every package in the tree onto the patched version. Around 59 of the 101 audit findings disappear in one command.

04 Scenarios closed

What an attacker could no longer do.

The same three concrete attack scenarios from the original briefing — re-evaluated against the post-fix codebase. Each one was easy on Monday and is now impossible (or vastly more expensive) on Friday.

01

The data leak Closed

The old version, in 10 minutes. Sign up for a free Kyozo account, query Firestore from the browser, walk away with every user's name, email, and phone number. Now. The Firestore rules deny that read; the most a fresh signup gets back is their own row. The workspaceMembers table — which used to leak the org chart — is similarly locked.

02

The takeover Closed

The old version, in 3 minutes. Type /sa in the URL bar, see a password prompt with the hint "platform name plus three numbers", try kyozo123, see every community in the system and reassign the largest one to yourself. Now. The page is deleted. The route is deleted. The hardcoded password no longer exists in the source tree. Workspace administration goes through the proper auth-gated flow.

03

The forged inbox Closed

The old version, in seconds. POST a JSON payload to /api/inbox/webhook, have it stored in Firestore as if Resend had delivered it, use that to phish a community owner via the in-app inbox. Now. The webhook verifies a Resend Svix HMAC signature with a five-minute replay window, refuses unsigned payloads in production, and writes only via Admin SDK against rules that no longer accept arbitrary creates.

06 Your action items

Three things only you can do.

The code changes are committed and reviewable. These three steps need a real engineer with dashboard access — they're not things I can complete from a sandbox. Each is straightforward; the runbook in docs/SECRET_ROTATION_RUNBOOK.md walks through the secret rotation in detail.

  1. Rotate every secret in the .env file

    Firebase service account, Resend API key, Gemini API key, GoDaddy API key & secret, the Vercel OIDC token. Each goes into Vercel's environment-variable store before the old key is deleted at the provider, so there's no outage window. Add a new WHATSAPP_WEBHOOK_SECRET and RESEND_WEBHOOK_SECRET at the same time — webhooks now require them in production.

    ~30 minutes
    docs/SECRET_ROTATION_RUNBOOK.md

  2. Run pnpm install on the security branch

    The sandbox disk was full so I couldn't run it for you. Once it runs against the updated package.json, the new pnpm.overrides entries pull every transitive dependency onto its patched version. This single command resolves about 59 of the 101 audit findings.

    ~5 minutes
    pnpm install & smoke-test

  3. Deploy the new database, storage, and CORS rules

    Run firebase deploy --only firestore:rules,storage:rules to push the tightened Firestore and Storage rules to production. Then re-apply the storage CORS configuration with the existing apply-cors.sh script. Smoke-test the apps after — invite a team member, approve a waitlist entry, upload a post image, sign in as a fresh user and claim a pending invite.

    ~10 minutes
    firebase deploy + smoke test

07 Going forward

What's still worth doing in the next quarter.

Four pieces of hardening we deferred from this sprint because they're more about preventing the next audit-cycle backlog than closing today's holes. None are urgent. All are worth a couple of engineering days each, and together they shift KyozoVerse from "responding to issues" to "automatically catching them before they ship."

Operational discipline

  • Dependabot or RenovateAutomatic PRs whenever a dependency CVE drops, instead of running npm audit by hand once a quarter. Most engineering teams set this up in 30 minutes and never think about it again.
  • Secrets out of .env, into a manager1Password Developer Tools or Doppler. Adds rotation reminders, audit logs, and per-engineer access control. Cheap insurance once the team grows past two people.
  • Re-enable TypeScript and ESLint in buildsThe Next config currently sets ignoreBuildErrors: true. Turning that off catches a class of bugs at compile time instead of in production logs.
  • Schedule a follow-up audit at the 90-day markThe whole point of fixing this sprint's findings is to see the next audit return a much shorter list. Put it on the calendar now while there's momentum.

Defence in depth

  • Security headers in next.configContent-Security-Policy, Strict-Transport-Security, X-Frame-Options. Catches an entire class of XSS, clickjacking, and protocol-downgrade attacks at the browser level. About one afternoon of work.
  • Persistent rate limitingThe current limiter uses an in-memory map per serverless instance, which is enough to catch casual scripts but not coordinated abuse. Upstash Redis (~$10/month) gives you proper per-user, per-route limits.
  • Logging & alerting on auth failuresSentry's free tier or similar. Set an alert for "more than 50 failed auth attempts in 5 minutes". Most attacks announce themselves in logs before they succeed, if anyone's watching.
  • DNSSEC on the kyozo.com domainOne checkbox in the GoDaddy dashboard. Closes a class of DNS-hijack attacks that would otherwise let someone re-issue TLS certs in your name.

08 In closing

The interesting thing
is what didn't need fixing.

The product architecture is sound. The data model is clean. The code is reasonable. What we found was the residue of moving fast — the prototype defaults that never got tightened, the dependencies that drifted, the route that was duplicated when work moved between branches. That residue is now cleaned up. The version of KyozoVerse on the security/critical-fixes branch is in good shape to scale.
