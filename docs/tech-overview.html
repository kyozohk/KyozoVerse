01 The shape of the thing

A serverless app on top
of managed services.

Kyozo is a Next.js application hosted on Vercel, talking to Firebase for authentication, database, and file storage, Resend for transactional and inbound email, GoDaddy for DNS-driven custom domains, and Google's Gemini for AI generation. There is no traditional backend server — every API endpoint is a serverless function, every database call goes directly from the browser or a function to Firebase. The thirty-thousand-foot picture:

Five managed services. No always-on backend. Every request handled by a Vercel serverless function or directly by a managed API.

The strength of this shape is that it scales itself — there's no server to size, no database to provision, no email infrastructure to maintain. The weakness is that we depend heavily on five vendors, and each one is a single-point-of-failure for whatever it owns. Section 9 returns to the trade-off.

02 Frontend

Modern, but not bleeding edge.

The frontend is built on the React + Next.js App Router stack — the same one Vercel, OpenAI, Hashnode, and most newer SaaS products use. We're on Next.js 15, React 18, Tailwind CSS, and Radix UI for accessibility primitives. TypeScript is enabled across the codebase. There's nothing exotic here, which is good: hiring is easy, the docs are excellent, and the patterns are well-understood.

Framework

Next.js 15 · App Router

The web framework that wraps React. App Router is the post-2023 way of building Next apps — server components, server actions, file-based routing. Mature, well-documented, the default choice for new React work.

03 Backend & infrastructure

Five vendors doing the heavy lifting.

There's no "Kyozo backend server" in the conventional sense. The application talks directly to managed APIs, and our serverless functions are mostly thin orchestration layers — accept a request, validate it, talk to one or two of the vendors, return JSON. Each vendor is paid per-use, so cost scales smoothly with usage rather than in step-function jumps when adding a server.

Database, auth, files

Firebase

The single biggest dependency. Firebase Auth handles signups and sessions; Firestore stores users, communities, posts, members, messages; Cloud Storage stores uploaded media. The Admin SDK lets server-side code bypass the client security rules. Generous free tier, predictable scaling, hard to misuse with the post-audit rules.

firebase ^11.9.1
firebase-admin ^13.6.0

Hosting

Vercel

Hosts the Next.js app. Each git push to main triggers a build; each pull request gets its own preview URL. Edge network, automatic HTTPS, environment variables managed in the dashboard. The trade-off is vendor lock-in to Vercel's specific Next.js features and pricing.

No npm package — handled at the platform level.

Email

Resend

Sends transactional and broadcast email and receives inbound replies. Modern, developer-friendly API, good deliverability. One key per environment, signed webhooks (now verified). We also have a per-community-domain pattern that depends on Resend's "add domain" API — see section 5.

resend ^6.4.2

DNS & domain

GoDaddy

Owns the kyozo.com domain and provides the DNS records that wire it together. Importantly, we drive GoDaddy's API programmatically when a community is created, automatically adding SPF, DKIM, and MX records for that community's subdomain. See section 5 for what happens when a community gets handle.kyozo.com.

No npm package — direct REST integration.

AI inference

Google Gemini · Genkit

Gemini Flash is the language model that powers the "improve this email" and AI assistant features. Genkit is Google's open-source orchestration library — it handles streaming, tool calling, and observability around model calls so we don't have to write that infrastructure ourselves.

@google/generative-ai ^0.24.1
genkit ^1.20.0
@genkit-ai/google-genai ^1.20.0
@genkit-ai/next ^1.20.0

Messaging (deferred)

360dialog · WhatsApp Deferred

Integration code exists for the WhatsApp Business API via 360dialog — broadcast templates, inbound webhook, signature verification. Not actively shipped or promoted. Kept as the foundation for the unified-inbox vision in section 6.

No npm package — direct REST integration.

The opinion. The backend choices are conservative and correct for the current scale. Firebase is the highest-leverage decision — it gives us a database, auth, file storage, and a generous free tier in one bill. The risk to flag is that a serious migration off Firebase would be expensive (rewriting security rules, re-implementing auth, moving objects) — so any plan to grow past Firebase's sweet spot needs to be started early.

05 Custom domain per community

Every community gets its own subdomain.

When a community owner creates aqua-splash, our system automatically provisions aqua-splash.kyozo.com as a real, working subdomain — capable of receiving email at reply@aqua-splash.kyozo.com and presenting a branded experience. This is one of the more distinctive technical choices in the product, and it's worth understanding what it gives us, what it costs, and where the risks live.

From the owner clicking "create community" to a working subdomain takes about 15 seconds and three API calls.

Why we do it

  • Each community feels independent — emails to its members come from aqua-splash@kyozo.com, not a shared no-reply address. Higher open rates, better deliverability.
  • The reputation of one community's mail can't poison another's. If a community owner triggers spam complaints, only that subdomain's reputation suffers.
  • Inbound replies route directly into that community's inbox. We can tell which community a reply belongs to from the recipient address alone.
  • The pattern extends naturally to white-label custom domains in the future ("send from community@mycoolclub.com") without rewriting the email layer.
  • SPF / DKIM / DMARC are configured automatically; community owners never see DNS.

What it costs us

  • The setup flow depends on three external APIs in sequence (Resend, GoDaddy, then back to Resend for verification). Any one of them being slow or down delays community creation.
  • DNS propagation typically takes seconds with GoDaddy but can stretch to minutes — owners occasionally see "domain not yet verified" briefly after creating.
  • The kyozo.com DNS zone gradually fills up with records. We're well within GoDaddy's limits, but at 10,000+ communities this becomes a record-management problem.
  • Renaming a community handle requires deleting the old DNS records and creating new ones. The renaming code path is 250+ lines and is a frequent source of bugs.
  • Couples us to GoDaddy's API specifically. Switching registrars would require rewriting that integration.

The opinion. The pattern is a net win at our current scale and a strategic differentiator — most competitors share a single mail domain across all their customers. The risk to plan for is the ten-thousand-communities case: at some point we'll want to either move to dynamic DNS records (Cloudflare's API is more permissive than GoDaddy's), or move to a model where small communities share a domain and only paid-tier communities get their own. Either is a quarter of work — easier to plan now than scramble later.

06 The unified inbox

One thread per member, across every channel.

A community owner today gets emails, will tomorrow get WhatsApp messages, will eventually get in-app messages from a mobile app, and in some cases gets SMS replies. The unified-inbox vision is that all of those land in a single thread per member — the owner reads the whole conversation in one place and responds without having to remember which channel the message came from.

All channels feed the same Firestore collection. Adding a new channel is a new webhook route, not a new data model.

Live today

Email (inbound and outbound) flows end-to-end through the inbox. When a member replies to a broadcast, Resend's inbound webhook fires /api/inbox/webhook, the route verifies the Svix signature, looks up the member by sender email, and writes a row into the inboxMessages collection. The community owner sees it appear in real time in the inbox UI, threaded with anything else that member has sent or received.

Outbound replies from the inbox UI go through /api/send-email, also storing the outgoing message in the same collection so the thread reads chronologically.

Coming next

WhatsApp is plumbed but not active. The webhook route, signature verification, and writes are all built — turning it on requires the WhatsApp Business API account from 360dialog and a product decision about whether to promote it.

Mobile app push would arrive as another webhook (or, more likely, a server action triggered by the app itself). The data model already supports it: messagingService: 'app' alongside 'email', 'whatsapp', etc.

SMS, Telegram, Slack-Connect-style integrations are all natural extensions of the same pattern. Each is one webhook route plus dashboard UI for the owner to enable it.

The thing this unlocks

A community owner can broadcast on email, get replies on WhatsApp, follow up via the mobile app, and never lose context — the thread reads as one conversation. Most platforms force the owner to track which channel a member uses; we plan not to.

07 AI features

Gemini, on tap, where it earns its keep.

The AI integration is intentionally narrow: where a human would obviously benefit from a draft, a model is offered. The premise is that an AI feature should save a community owner time on something they were going to do anyway, not invent work. Today that means polishing email subjects, drafting broadcast bodies, and answering "how do I do X in this product" questions inside the dashboard.

A simple pipeline today; the right place to grow it is where users actually ask for more help.

Email subject polish Live

The subject input field on broadcast and invite forms has a sparkle button. Click it, the model sees the rest of the context (community name, recipient list, body draft) and returns a shorter, livelier subject. Fastest-path AI feature — high frequency, immediately useful.

Broadcast body drafting Live

Same UI pattern on the body textarea. Generates an email or WhatsApp template draft from a prompt the owner provides ("Welcome new members and tell them about Saturday's event"). The owner edits and sends; the model is a starting point, not an autopilot.

In-dashboard AI assistant Live

The community overview has a chat box: "How do I add a tag to multiple members?" The model answers in plain English about the product. Currently a hand-written prompt, no retrieval — but a clear path to upgrade with RAG over our docs.

Member-list summaries Next

"Show me which members haven't posted in 30 days but were active before that" — natural-language filtering of the member list. Requires giving the model tool access to the member-query API. Genkit makes this two days of work, not two weeks.

Auto-reply suggestions Next

When a member replies in the inbox, the model proposes 2–3 short replies the owner can pick from. Same pattern as Gmail's smart replies. Inbox UI is already the right surface for this; the model has the thread context.

RAG over community content Future

Members ask questions, the model answers based on the community's own posts and pinned content. Worth building once communities have enough content to make it a real time-saver — probably six months out. Vector storage in Firestore (or Vertex AI Vector Search) is the obvious path.

The opinion. The right way to grow this is to watch what users do with the existing AI buttons and add the next feature where the data says they want it. Genkit gives us streaming, tools, and traces effectively for free, so the marginal cost of adding an AI feature is small once the orchestration is in place. The risk is the cost line: at scale, AI inference is the only piece of infrastructure where bills can grow non-linearly. Worth setting per-community budgets early.

08 What's coming next

Five capabilities worth a plan.

Beyond the current product, the technical foundation is well-positioned for several specific extensions. Each of these is a known roadmap item — not speculative — and each builds on infrastructure that's already in place.

QR-code member onboarding Next quarter

An owner shows a QR code at an event; an attendee scans it with their phone camera; they're taken to a pre-filled signup form for that specific community, opt in to email or WhatsApp, and are added as a member without typing anything. The QR encodes a signed token tied to the community handle and a short-lived expiry.

Pieces involved: a signed-token-mint endpoint, the QR rendering on the owner dashboard, a public landing page that consumes the token, and the existing waitlist / member-creation flow as the destination.

Time: ~2 engineering weeks. Why it matters: in-person events are how most communities really grow, and the friction at signup tables today is enormous.

Public API · v1 Started on API_V1 branch

An external API (under /api/v1/) so community owners can integrate Kyozo into their own tooling. The branch API_V1 already has the foundation: API-key auth (kyz_ prefixed keys, hashed at rest), scope system (email:send, communities:read, etc.), and the first three endpoints. Pulling this onto main is the next step.

What it unlocks: Zapier integrations, custom dashboards, programmatic member imports, embedding Kyozo in a customer's own product. Also the foundation for the MCP server below.

Time: ~3 weeks to merge, document, and ship. Plus: an OpenAPI spec, hosted docs, and rate limits per key.

MCP server After v1 ships

Once the public API is solid, exposing it as a Model Context Protocol server lets ChatGPT, Claude, Cursor, Cowork, and other AI clients work with a community owner's Kyozo workspace as a tool. "Send a welcome email to new members from yesterday" becomes a single sentence in the user's preferred AI assistant.

Pieces involved: a thin MCP wrapper around the v1 API, OAuth or API-key bridging to the user's Kyozo identity, hosting (likely the same Vercel functions). MCP TypeScript SDK is already in the dependency tree.

Time: ~1 week once v1 is stable. Why now: early MCP adoption gives Kyozo presence in the AI tool directories most users will spend the next year inside.

Mobile app 2026 H2

A React Native (or Expo) app for community members. Read posts, get notifications, reply in the unified inbox, scan QR codes, view events. The app shares the existing API surface — the same endpoints the web app uses can serve a mobile client with no changes.

Why this shape: existing scaffolding for it (/api/mobile/* routes for push tokens) is already laid down. Firebase Auth has native iOS/Android SDKs that pair cleanly with the same identity model.

Time: ~3 months for an MVP. Decision: whether to lead with a member app or an owner app; we recommend member-first because that's where the daily-active surface lives.

Owner-facing analytics & insights 2026 H2

Engagement charts already exist; the next layer is "what's actually working." Most-engaged members, post types that drive replies, broadcast send-time optimization, member-churn alerts. Mostly aggregations on top of the data we're already collecting — no new data sources needed.

Pieces involved: scheduled functions to compute aggregates, a few new collections, and chart components built on Recharts (already in the stack).

Time: ~4 weeks for v1. Recommendation: ship the rough version early and iterate — the right metrics aren't obvious until owners use them.

Other community-owner wins Backlog

Smaller items each owner asks for once: scheduled posts, cross-community broadcasts (for owners running multiple), Stripe-style subscription billing for paid memberships, custom-domain white-labelling beyond {handle}.kyozo.com (e.g. community.theirbrand.com), Eventbrite ticket sync, Zoom integration for events. Each is one or two engineering weeks if prioritised.

09 Recommendations & roadmap

A 12-month technical horizon.

Tying everything together: the roadmap below sequences the items most worth doing, ordered by a rough "user-visible impact divided by engineering cost". Some are reliability work, some are growth features. Both kinds matter.

Now

Land the security sprint Shipped, awaiting deploy

Merge security/critical-fixes into main, rotate the secrets per the runbook, deploy the new Firestore rules, run pnpm install. This unblocks any further changes from getting tangled in the same files.

Month 1

Merge API_V1 into main

Take the public-API foundation out of its branch and into the main product. Document the v1 endpoints, set up rate limits per API key, and write the first integration partner-facing docs.

Month 1

QR member onboarding

Highest-leverage owner-facing feature on the list. Two engineering weeks if the team focuses; opens the in-person growth channel that the product currently has no story for.

Month 2

Defence-in-depth hardening

Re-enable TypeScript and ESLint in builds, add the security headers (CSP, HSTS, X-Frame-Options) to next.config.ts, replace the in-memory rate limiter with Upstash Redis, set up Sentry for failed-auth alerting. Each item is a half-day of work; collectively they're a major improvement.

Month 2–3

MCP server v1

Wrap the v1 API as an MCP server so AI clients (Claude, ChatGPT, Cursor, Cowork) can drive Kyozo workspaces. Distinguishing factor in a market where every community platform looks the same on the surface.

Month 3–4

Auto-reply & inbox AI

Smart-reply suggestions in the inbox, member-list natural-language search, auto-summarise long threads. All build on the existing Genkit + Gemini plumbing.

Month 4–6

Mobile app · MVP

React Native / Expo app for members. Push notifications, post reading, inbox replies, QR scanning. Three months to MVP if dedicated; longer if shared with web work.

Month 6–9

Activate WhatsApp inbox

The integration is built; what's needed is the 360dialog Business account, the product positioning, and a rollout plan. WhatsApp is where most community conversation actually happens internationally — turning this on is a meaningful product expansion.

Month 9–12

Owner analytics, RAG, scheduled posts

The second wave of features once the platform shape is set: deeper insights for owners, member-facing AI Q&A grounded in the community's own content, scheduled and recurring posts, Stripe-style paid memberships.

Continuous

Hygiene that should never stop

Dependabot/Renovate auto-PRs for CVE updates. Quarterly secret rotation. Annual external security audit. Performance budget on the Vercel build. None of these are urgent; all of them prevent next year's audit being as long as this year's.

Three strategic recommendations

  1. Treat the v1 API and the MCP server as one product release

    Most of our competitors don't have either. Shipping them together — plus the QR onboarding — is a coherent "Kyozo for the AI era" story rather than a series of feature drops. It also makes the API documentation effort pay for itself across both surfaces.

    Product positioning

  2. Decide on the mobile-app order of operations now

    The scaffolding is there but the strategic question (member-first vs. owner-first) drives different UX investments. Member-first builds daily-active engagement and creates an obvious upsell path; owner-first deepens the existing customer relationship. Recommendation: member-first, but the decision changes the next six months of roadmap.

10 In closing

Boring choices in the
foundation, interesting ones
where it matters.

The technology behind Kyozo is intentionally unflashy in the parts that have to work — Next.js, React, Firebase, Vercel, all proven and well-staffed. The interesting decisions are higher up: a custom domain per community, a unified inbox across email and (eventually) every other channel, a public API and MCP server that turn Kyozo into a building block other tools can use. That layering is what separates a defensible product from a thin CRUD app, and it's the part most worth investing in next.

Questions, push-back, alternative plans — let's discuss.
